# 逐章逐單元逐題原稿答案校對流程規格

- 制定日期：2026-07-16
- 適用範圍：`C:\insurance_ai` 保險題庫之答案校驗、錯題登記、修正準備、批次修正與修正後驗證
- 文件定位：後續所有答案校驗、錯題登記與批次修正的最高流程依據

## 一、文件目的

本文件規定保險題庫答案校驗的正式流程。所有題目必須依原稿檔案、章、單元及題號，逐題回到原稿 PDF／Word 比對題幹、選項、答案及解析；不得以 SQLite、JSON 或 AI 推論作為答案正確性的最終依據。

本流程的目的，是建立可重現、可追溯、可人工複核的證據鏈，並在正式修改題庫前清楚區分原稿內容、系統現況、差異類型及建議修正內容。

## 二、最高原則

1. 原稿 PDF／Word／官方法規來源為最高依據。
2. SQLite 是 Web 系統正式執行資料，但不是答案正確性的最終依據。
3. `all_questions.json` 是正式題庫快照，但可能繼承轉換錯誤。
4. `output` JSON 是中間來源或匯入來源，不得直接視為正確答案。
5. AI 與程式只能輔助尋找疑點、建立對應及整理差異，不能直接決定答案。
6. 每一題正式修正前，必須有可追溯的來源頁碼、題號或法規依據。
7. 原稿本身若有疑義，不得逕行猜測；應標記為 `source_unclear` 或 `manual_review`，等待人工確認。
8. 題目、選項、答案與解析必須分欄比對，不得只比答案欄。

## 三、標準校對單位與順序

後續校對必須依下列層級執行：

1. 先依原稿檔案建立校對批次。
2. 再依原稿章別分組。
3. 再依章內單元分組。
4. 再依原稿題號逐題校對。
5. 最後對應 SQLite `questions.id` 與 `all_questions.json` 的 `id`。

不得先以 SQLite ID 順序取代原稿順序。若一題對應多個系統 ID、同一系統 ID 疑似對應錯誤來源，或改寫題與原題並存，必須分別建檔並標示映射關係。

## 四、每題校對欄位

每一題至少必須記錄下列欄位：

| 欄位 | 說明 |
|---|---|
| `audit_id` | 本次校對紀錄的唯一識別碼 |
| `source_file` | 原稿檔案路徑或官方來源名稱 |
| `source_page` | 原稿頁碼；應註明 PDF 頁碼或印刷頁碼基準 |
| `source_chapter` | 原稿章別 |
| `source_unit` | 原稿單元 |
| `source_question_no` | 原稿題號 |
| `source_question_text` | 原稿題幹全文 |
| `source_options` | 原稿選項，保留原順序與代號 |
| `source_answer` | 原稿答案 |
| `source_explanation` | 原稿解析；無解析時明確標記空缺 |
| `sqlite_id` | 對應的 SQLite `questions.id` |
| `json_id` | 對應的 `all_questions.json` ID |
| `system_question_text` | 系統目前題幹 |
| `system_options` | 系統目前選項 |
| `system_answer` | 系統目前答案 |
| `system_explanation` | 系統目前解析 |
| `answer_match` | 原稿與系統答案是否一致 |
| `option_match` | 原稿與系統選項是否一致 |
| `explanation_match` | 原稿與系統解析是否一致 |
| `issue_type` | 一項或多項錯誤類型代碼 |
| `confidence` | 來源映射與判定信心程度，並附判定依據 |
| `reviewer_note` | 人工複核備註 |
| `fix_suggestion` | 建議修正內容；不得在校對階段直接套用 |
| `status` | 本題目前流程狀態 |

布林比對欄位如無法判定，不得強制填入 true 或 false，應搭配 `source_unclear` 或 `manual_review` 說明原因。

## 五、狀態代碼

| 狀態 | 定義 |
|---|---|
| `pending` | 尚未開始來源追蹤或校對 |
| `source_found` | 已找到可能原稿，但尚未完成逐欄比對 |
| `matched` | 題幹、選項、答案及必要解析均與原稿一致 |
| `mismatch` | 已確認至少一個比對欄位不一致，尚未完成錯誤分類 |
| `ocr_parse_error` | OCR、PDF 表格抽取或文字解析造成結構錯誤 |
| `option_pollution` | 選項混入題幹、解析或相鄰欄位內容 |
| `answer_wrong` | 已依原稿確認系統答案錯誤 |
| `explanation_pollution` | 解析被污染，或被併入題幹／選項 |
| `source_unclear` | 原稿不清楚、頁碼不明或來源互相衝突 |
| `manual_review` | 必須由人工目視或專業人員判斷 |
| `ready_to_fix` | 證據完整、修正內容明確，已可進入修正清單 |
| `fixed` | SQLite／JSON 等正式資料已依核准方案修正 |
| `verified` | 修正後已完成資料一致性與 Web 顯示驗證 |

單題可有一個主要狀態及多個 `issue_type`。狀態必須依實際進度更新，不得在尚未修正時標記為 `fixed`，也不得在未驗證時標記為 `verified`。

## 六、錯誤類型

| 錯誤類型 | 定義 |
|---|---|
| `wrong_answer` | 答案欄位與原稿不一致 |
| `option_pollution` | 選項混入解析文字、題幹或相鄰資料 |
| `explanation_missing` | 原稿有解析，但系統解析缺漏 |
| `explanation_pollution` | 解析被併入選項或題幹，或混入其他內容 |
| `truncated_question` | 題幹遭截斷 |
| `truncated_explanation` | 解析遭截斷 |
| `duplicate_conflict` | 同題或高度相似題存在不同答案，且無法由格式正規化解釋 |
| `source_mapping_error` | 系統題目對應到錯誤原稿、頁面、題號或改寫版本 |
| `outdated_law` | 教材答案與現行法規間存在時效或版本疑義 |
| `manual_review` | 無法可靠自動判斷，需人工目視或專業判斷 |

同一題可以同時包含多種錯誤。例如解析被併入第 4 選項且答案誤抓時，應同時記錄 `option_pollution`、`explanation_pollution` 與 `wrong_answer`。

## 七、第一批試跑範圍

第一批試跑不得直接涵蓋全題庫，指定範圍如下：

- 原稿：`input/JY-人身保險.pdf`
- 章節／單元：三、保險費架構、解約金、準備金、保單紅利
- 頁碼範圍：以 Codex 後續唯讀盤點確認

選擇此範圍的原因：

1. ID 3815 已確認該單元存在轉換錯誤。
2. PDF、raw text、`output` JSON、`all_questions.json`、SQLite 的來源鏈清楚。
3. 此單元同時涵蓋題幹、選項、答案及解析欄位，適合作為逐題校對流程的試跑樣本。
4. 先以受控單元驗證映射規則、狀態轉換及輸出格式，可降低直接處理全題庫造成的誤修風險。

## 八、第一批試跑應輸出

後續第一批唯讀試跑任務必須輸出：

1. 該單元原稿題號清單。
2. 每題對應的 SQLite ID。
3. 每題對應的 `all_questions.json` ID。
4. 原稿答案與系統答案對照表。
5. 原稿選項與系統選項對照表。
6. 原稿解析與系統解析對照表。
7. 不一致題目清單。
8. 無法自動對應題目清單。
9. 建議人工目視清單。
10. 明確聲明不得在試跑任務中直接修題庫。

試跑結果應保留來源頁碼、題號、比對方法及信心程度，使人工複核者不必重新猜測來源鏈。

## 九、與 ID 3815 案例的關係

ID 3815 是第一個已由人工目視確認的案例：

- 來源：`input/JY-人身保險.pdf`
- 頁碼與題號：P.89 第 18 題
- 原稿答案：`1`
- 系統答案：誤為 `3`
- 其他問題：第 4 選項被解析文字污染

此案例證明，系統答案、JSON 快照或轉換後解析結果可能共同繼承同一轉換錯誤。後續必須採用原稿逐題校對，不得單純相信系統答案或以多個衍生資料來源彼此相同作為正確性證明。

ID 3815 的正式個案紀錄見：

`docs/corrections/id3815_jy_p89_term_insurance_answer_correction_20260716.md`

## 十、後續正式流程

1. 建立本流程規格文件。
2. 建立錯題修正總表（correction ledger）。
3. 對第一批 JY 單元執行唯讀逐題校對。
4. 人工目視複核不一致題及無法自動對應題。
5. 將確認錯誤題登記到 correction ledger。
6. 累積並審查 `ready_to_fix` 清單。
7. 備份 SQLite 與 `all_questions.json`。
8. 依核准清單批次修正，並維持 SQLite 與正式 JSON 一致。
9. 啟動 Web 系統抽查題幹、選項、答案及解析顯示。
10. 檢查修正 diff 後建立 commit。
11. 經確認後 push。
12. 完成本批驗證後，再進入下一章或下一單元。

每一階段均須保留清楚的輸入、輸出及狀態，不得跨過人工複核、備份或驗證步驟。

## 十一、禁止事項

1. 不得未查原稿就修改答案。
2. 不得只因 AI 推論就修改答案。
3. 不得只因 SQLite 與 JSON 不一致就判定答案錯誤。
4. 不得直接用 `output` JSON 覆蓋正式題庫。
5. 不得未備份就修改 SQLite。
6. 不得未備份就修改 `all_questions.json`。
7. 不得把 ID 2670 與 ID 3815 這類改寫題混為同一題。
8. 不得在法規時效題中直接以現行法覆蓋原教材答案；必須標示教材版本、法規版本、生效日期及採用依據。
9. 不得以題幹文字相似作為唯一映射依據；還須核對選項、章節、題號及來源頁面。
10. 不得在唯讀校對階段直接修正 SQLite、JSON 或平台程式。
